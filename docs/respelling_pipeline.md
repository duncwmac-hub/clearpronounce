## ClearPronounce respelling and unlocker pipeline

This document summarizes the current end-to-end flow for generating Polish-like respellings from English words and analysing them against the Polish phoneme inventory.

### Data flow overview

1. **Inputs**
   - CMU Pronouncing Dictionary (CMUdict) text file.
   - Frequency list CSV with `word,frequency_rank`.
   - Phoneme inventories:
     - `rules/phonemes_english.md` (E).
     - `rules/phonemes_polish.md` (P).
   - English → Polish respelling rules: `rules/english_to_polish_respellings.md`.

2. **Ingestion**
   - `lexicon/ingest_cmudict.py`:
     - Loads CMUdict and the frequency list.
     - Uses `ARPABET_TO_IPA` and `arpabet_to_ipa` to convert ARPABET pronunciations into IPA strings consistent with the English inventory.
     - Computes a simple set-style `phoneme_set` from the IPA string.
     - Calls `generate_respelling(ipa)` from `lexicon/respeller.py` to produce a canonical Polish-like respelling for each word.
     - Writes a lexicon CSV with columns:
       - `word`
       - `ipa`
       - `cmu_pron`
       - `phoneme_set`
       - `polish_respellings`
       - `pos`
       - `frequency_rank`

3. **Respelling rules**
   - `rules/english_to_polish_respellings.md` is the human-edited source of truth.
   - `lexicon/respelling_rules.py` parses the markdown table into a dictionary:
     - Keys: IPA phonemes (e.g. `tʃ`, `ə`, `eɪ`).
     - Values: `RespellingRule` objects with:
       - `english_phoneme`
       - `default_letters` (single canonical mapping)
       - `notes`

4. **Respeller**
   - `lexicon/respeller.py` provides:
     - `tokenize_ipa(ipa: str) -> list[str]`:
       - Splits the IPA string on spaces.
       - Assumes affricates and diphthongs are already single tokens (as produced by `arpabet_to_ipa`).
     - `generate_respelling(ipa: str) -> str`:
       - Loads the respelling rules.
       - Maps each phoneme to its `default_letters`.
       - Falls back to the raw phoneme string if a mapping is missing.
       - Concatenates all pieces into a single respelling string (e.g. `/ð ə/` → `ðə`).

5. **Unlocker analysis**
   - `lexicon/unlocker.py`:
     - Reads a lexicon CSV generated by the ingest script (or a compatible seed lexicon).
     - Loads E and P from their markdown inventories via `load_inventory`.
     - For each word:
       - Parses `phoneme_set`.
       - Computes:
         - `already_pronounceable`: whether the word uses only phonemes from P.
         - `unlocked_by`: which new English phonemes in `E \ P` would make the word pronounceable if added.
     - Writes out a report CSV with the original lexicon columns plus:
       - `already_pronounceable`
       - `unlocked_by`

### Design principles captured in the current implementation

- **Intelligible to English listeners**
  - Diphthongs and many vowels use familiar Latin letter combinations (`ej`, `aj`, `oj`, `au`, `oł`).
  - Consonants mostly mirror English where that does not conflict with Polish (`p, b, t, d, k, g, f, s, z, h, m, n, l, r`).

- **Familiar and articulatory-friendly for Polish learners**
  - Polish-specific letters and digraphs are used where they are already part of the learners' orthographic system (`ć, ś, ż, ł`).
  - `/v/` is written as `w`, in line with Polish spelling.

- **New target sounds not approximated away**
  - `/θ, ð, ŋ, ə/` are kept as distinct symbols (`θ, ð, ŋ, ə`) and treated as genuinely new targets to learn, not replaced with approximate Polish phonemes.

### Future backend integration

When you are ready to expose this as a service under `src/`, the core building blocks are already aligned:

- A backend endpoint can:
  - Accept a word.
  - Look it up in the lexicon (or derive it on the fly via CMUdict).
  - Return:
    - IPA.
    - Polish-like respelling.
    - Unlocker information (already pronounceable vs unlocked-by).

This is enough to power curriculum generation, worksheets, or interactive tools for Polish learners working on English pronunciation.
